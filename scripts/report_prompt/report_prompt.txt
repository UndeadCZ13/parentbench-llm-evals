# 【一、极其重要的硬性写作要求（必须严格遵守）】

## 1. 全文字数与展开深度
- 报告总长度不得少于 **4500 中文字**
- 优先目标：**5500–7500 字**
- 宁可写得详细、系统、重复解释，也不要省略分析过程

## 2. 章节完整性
- 必须严格按照我指定的章节结构输出（见下文）
- 每一个一级章节：
  - 至少 **3 个自然段**
  - 每段不少于 **120 字**
- 禁止“只有一段”的章节

## 3. 证据驱动原则
- 任何结论必须基于我提供的证据材料：
  - 表格（stats / merged / scores）
  - 图像（radar / boxplot / winrate / heatmap / embedding 等）
- 每一个重要判断必须至少引用：
  - 1 份表格证据
  - 1 条 vision 模型生成的图像解读

## 4. 禁止图注式总结
- 你不能只描述“图中显示了什么”
- 你必须解释：
  - 为什么会出现这种现象？
  - 这反映了模型在 ParentBench 中的哪类能力？
  - 对工程部署意味着什么？

## 5. 能力 vs 工程稳定性区分
必须明确区分并反复强调：
- 模型能力不足（capability limitation）
- 工程/生成失败（空输出、截断、retry）
- 高分 ≠ 稳定 ≠ 可部署

# 【二、模型主体性分析（强制要求）】

## 6. 逐模型分析是强制的（Model-by-Model）
- **每一个被评测模型都必须被单独分析**
- 每个模型至少有一个完整分析段落
- 禁止只在总表或总结中顺带提及模型名称

## 7. 模型分组与类型化（Model Taxonomy）
你必须基于实验结果，对模型进行**功能性分组**，例如（仅为示例）：
- 高分且稳定模型
- 高分但波动大的模型
- 情感/沟通能力突出模型
- 语言鲁棒性强 vs 语言漂移明显模型

分组标准必须清晰、可追溯，并基于数据证据。

## 8. 重点模型深度对比
- 从所有模型中选出 **3–6 个关键模型**
- 在以下维度进行深入横向对比：
  - empathy / communication / safety / completeness
  - EN vs ZH 差异
  - 稳定性与方差
- 必须回答“为什么它们不同”，而不仅是“谁分高”

Scenario → Generation → QC → Judging → Aggregation → Analysis
